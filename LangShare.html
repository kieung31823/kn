Language Sharing Interface


Project Link

Introduction

When automatic speech recognition systems appeared to low efficacy, human intervention is neccessary for transcription of endangered languages. This is because lack of data in knowledge domain or low quality of recordings does not allow the creation of reliable speech recognition system.
While some challenges exists, such as time consuming, limited linguists and Indigenous speakers; there has been a demand for tools enabling the language sharing between Indigenous speakers and linguists to support transcription works.

Description

This project is to build an interface assisting the language verification in recordings (manual transcription) between Indigenous speakers and linguists. The interface enables Indigenous end-users to play and hear the recordings, then verify words existing in the recordings.

Objectives

  • Transcription interface support the manual works
  • Collecting standard inputs for automatic speach recognition machine.

Scopes

  1. Research and design the solution
  2. Developing the interface (app) solution for Indigenous users.
  3. Testing with Indigenous speakers

Testing

TDD Unit Testing

Test-Driven Development (TDD) is used as a practice in the development process to ensure the program meets requirements, preventing big bug and ensure the possible solution is the minimum amount of code to pass the test




Integration Testing

To test the system modules after they are integrated, End-to-End testing used the Jest testing framework is run to test the functionalities of system before each version release for client testing.

3 Cycles of PAR

- The testing participants: Kunwinjku speak including males and females over 18 years old in Northern Territory, Australia.
- The testing tool: the developed interface.
- The participatory action research (PAR) is selected to test the effectiveness of development, because of the short development period and limited availability of participants.

A cycle of PAR includes four steps:
  1. Diagnosing problems through the tests with end-users (Indigenous participants)
  2. Collecting testing data
  3. Sharing data through stakeholders
  4. Taking actions or reflections on the problem.
The App is tested with two Indigenous participants though the cognitive walkthrough testing. Cognitive walkthrough is suitable to test interface with Indigenous people who have little experience about the software. Participants are discussed about the target of tasks, then they will explore the interface and solve the tasks. With each cycle, problems are explored and actions are taken (redesigned).

Result & Reflection

Result:
The developed application has now been used by Top End Language Lab in field work, for supporting the transcription tasks between Aboriginal and non-Aboriginal people

Found Technical Limits:
- Drag and drop gestures with Vue instance Lifecycle: With drag and drop function, the syllable data changes but no changes of DOM elements. The application is run with a single Vue instance following by a lifecycle of hooks from Initiating Vue Instance, to create, to mount and after that. The difficulty is the storing of sorted syllables (temporary data) in array, Vue instance cannot watch their changes and not produce DOM element for new added syllables. The solution is using the watcher to response to any changes of data in temporary array to evoke the refresh of the mount in Vue and create DOM for new sorted syllable data.
- Lack of support for vue in Ionic framework: Lack support of V-model in ion-search input. It required to use HTML or JavaScript code as the substitute.
- Python script cannot be executed by the fontend framework, it is required backend-servers and as the result of that, the system deployment becomes a challenge for real-time execute the python script. Another issue is the slow process of python script to the voice recording file. That impacts on the performance of the respeaking function when user records a long speech.
